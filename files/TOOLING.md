# Default agent tooling (AXI CLIs)

Three AXI CLIs are the system default for their domains. Each default holds unless I explicitly opt out for that request (e.g. "don't use lavish", "don't use gh-axi", "don't use chrome-devtools-axi", or an equivalent) - an opt-out applies only to the current request or task, and the default is back on for the next one. If a default tool genuinely cannot do what is needed, fall back to the alternative and say so.

- Plans and docs -> Lavish (`lavish-axi`). Any plan or document I would review - implementation plans, design docs, technical proposals, option comparisons, reports, analyses - must be delivered as a rich, reviewable Lavish HTML artifact instead of markdown prose. This includes plan mode: when presenting a plan for approval, build it as a Lavish artifact rather than terminal markdown. Follow the `lavish` skill workflow (generate the HTML, open it with `lavish-axi <file>`, then poll for feedback). Short answers, quick replies, and single-point responses stay as normal chat - this is about reviewable deliverables, not every message.
- GitHub -> `gh-axi`. For GitHub collaboration and API work - issues, pull requests, CI/Actions runs, releases, and repo/label/secret/variable administration - use `gh-axi`, not raw `gh`, `gh api`, or the GitHub REST/GraphQL API directly. Plain git transport (clone, push, pull, fetch) stays plain `git`. Fall back to raw `gh` only when `gh-axi` has no command for the operation.
- Browser -> `chrome-devtools-axi`. For any browser work - debugging a running page (console, network, DOM), plus automation, navigation, inspection, screenshots, and reproducing UI bugs - use `chrome-devtools-axi`, in preference to Playwright, puppeteer, or any Chrome DevTools MCP.
